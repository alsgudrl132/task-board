# 설계 결정 (DECISIONS)

## 1. 상태 구조

서버 상태와 클라이언트 상태를 세 개의 ref로 분리해 관리했다.

- `tasksRef` — 현재 화면에 보이는 최신 tasks. 낙관적 업데이트가 즉시 반영된다.
- `savedTasksRef` — 서버가 마지막으로 성공 확인한 tasks (Map<id, Task>). 실패 시 롤백 기준점.
- `queueRef` — task별 요청 Promise 체인. 경쟁 상태 방지용.

`useState`만으로는 비동기 콜백 내에서 최신 상태를 읽을 수 없다(클로저 문제). `tasksRef.current`를 항상 최신으로 유지해두면 큐에서 다음 요청이 실행될 때 올바른 version 번호를 읽을 수 있다.

고려했다가 버린 대안: Redux/Zustand 같은 외부 상태 관리 라이브러리. 이 과제에서 상태 구조가 복잡하지 않아 도입 비용 대비 이득이 없다고 판단했다.

## 2. 낙관적 업데이트 & 롤백

모든 쓰기 작업(이동·수정·삭제·생성)은 서버 응답을 기다리지 않고 UI를 먼저 반영한다.

- **이동**: `tasksRef`에 즉시 반영 → 서버 요청 → 성공 시 version만 동기화, 실패 시 `savedTasksRef`에서 복원
- **수정**: 원본 task를 `target`에 저장 → UI 즉시 반영 → 실패 시 `target`으로 복원
- **삭제**: 목록에서 즉시 제거 → 실패 시 다시 추가
- **생성**: 임시 ID로 먼저 추가 → 서버 성공 시 진짜 ID로 교체, 실패 시 임시 항목 제거

롤백은 **이전 스냅샷 복원** 방식을 택했다. 서버 재조회는 추가 네트워크 비용이 들고, 실패 상황에서 또 실패할 가능성이 있다. 로컬에 저장해둔 마지막 서버 확인 상태로 즉시 되돌리는 것이 더 안정적이다.

이동 성공 시 `serverTask`로 통째로 교체하지 않고 version만 동기화하는 이유: 연속 이동 중 첫 번째 응답이 오면 서버의 status(이미 지난 상태)로 덮어써서 화면이 깜빡이는 버그가 발생했다. 현재 낙관적 status를 유지하고 version만 올리면 이 문제가 해결된다.

## 3. 경쟁 상태(race condition)

task별 Promise 큐(`queueRef`)로 요청을 직렬화했다.

```
이동 A→B: 큐에 요청1 추가 (즉시 실행)
이동 B→A: 큐에 요청2 추가 (요청1 완료 후 실행)
요청1 완료 → tasksRef에서 최신 version 읽어 요청2 전송
```

핵심은 요청2가 큐에 등록되는 시점이 아니라 **실행되는 시점**에 `tasksRef.current`에서 version을 읽는다는 것이다. 등록 시점에 읽으면 요청1 완료 전이라 version이 구버전이고, 서버에서 409 충돌이 발생한다.

고려했다가 버린 대안: 요청 취소(AbortController). 이전 요청을 취소하고 최신 상태만 전송하면 간단하지만, 서버가 중간 상태를 저장하지 못해 version이 어긋날 수 있다. 큐 방식이 최종 상태와 version 정합성을 모두 보장한다.

## 4. 대량 데이터 성능 (5,000개)

`@tanstack/react-virtual`로 가상 스크롤을 구현했다. 화면에 보이는 항목만 DOM에 렌더링하고 나머지는 빈 공간으로 처리한다.

카드 높이가 설명(description) 유무에 따라 달라지므로 `measureElement`를 사용해 각 카드의 실제 높이를 측정한다. 고정 `estimateSize`만 쓰면 설명이 있는 카드의 수정/삭제 버튼이 잘리는 문제가 발생했다.

포기한 것: 검색/필터 기능. 가상 스크롤 구현 후 시간 제약으로 Priority 2 항목인 검색·필터는 구현하지 않았다.

## 5. 정답이 없던 결정들

**네트워크 완전 단절 시**

요청 실패 시 alert로 사용자에게 알리고 이전 상태로 롤백한다. 별도의 오프라인 감지(navigator.onLine 등)는 구현하지 않았다. 과제의 핵심인 실패 롤백 자체가 단절 상황도 동일하게 처리하기 때문이다.

**409 충돌 시**

현재는 일반 실패와 동일하게 롤백 후 alert 처리한다. 이상적으로는 서버의 최신 상태(`ApiError.payload.current`)를 읽어 충돌 해소 UI를 보여줘야 하나, Priority 2 항목으로 시간 제약상 미구현이다.

**실패한 쓰기의 재시도**

수동 재시도를 택했다. 자동 재시도는 사용자가 의도적으로 취소한 작업을 모르는 사이에 재전송할 수 있고, 네트워크 문제가 지속될 경우 요청이 쌓인다. alert로 실패를 알리고 사용자가 직접 다시 시도하게 하는 것이 더 예측 가능한 동작이다.
